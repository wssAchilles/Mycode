\begin{center}
\texttt{http://localhost:8080/api/space/feed?limit=10\&in\_network\_only=false}
\end{center}

\section{数据库检查}

通过Supabase MCP读取数据库结构，确认项目\texttt{telegram}处于健康可用状态，数据库主机为Supabase托管PostgreSQL。核心业务表均已存在，且具备推荐系统所需数据基础。

数据库检查结果如下：

\begin{enumerate}
  \item 用户与社交关系数据充足。\texttt{users}表有642条记录，\texttt{group\_members}表有764条记录，\texttt{contacts}表有195条记录，可支撑社交召回。
  \item 内容数据具备推荐基础。\texttt{news\_articles}表有838条记录，包含标题、摘要、分类、关键词、向量和互动计数等字段。
  \item 行为数据可用于排序。\texttt{news\_user\_events}表有2386条记录，事件类型包括曝光、点击、停留和分享。
  \item 用户兴趣向量表已存在。\texttt{news\_user\_vectors}表保存短期向量和长期向量，可用于个性化匹配。
  \item 实验配置表已存在。\texttt{experiments}表有2条记录，可用于A/B测试和推荐策略灰度。
\end{enumerate}

\section{推荐算法测试}

推荐算法测试重点不是单个函数是否返回固定值，而是检查推荐流水线是否能覆盖不同用户状态。测试设计如下：

\begin{enumerate}
  \item 冷启动用户测试：当用户没有足够行为时，系统应减少兴趣向量依赖，优先使用热门内容、语言地区匹配和冷启动召回。
  \item 稀疏行为用户测试：当用户有少量关注和点击记录时，系统应同时使用社交图谱召回和热门召回，避免结果过窄。
  \item 活跃用户测试：当用户有较多行为事件和兴趣向量时，系统应提高向量召回、作者亲和和内容质量得分的影响。
  \item 负反馈测试：当用户屏蔽作者、标记不感兴趣或已曝光某内容时，过滤阶段应将其剔除或显著降权。
  \item 多样性测试：当多个候选来自同一作者或同一主题时，TopK选择阶段应控制重复度。
\end{enumerate}

\section{Go投递链路测试}

Go侧测试重点是验证消息投递链路在并发优化后仍保持业务顺序、失败语义和可观测字段。测试设计如下：

\begin{enumerate}
  \item keyed ordering测试：构造多个\texttt{chat\_id}和\texttt{message\_id}的Redis Stream消息，验证同一业务键进入同一worker lane并保持处理顺序，不同业务键可以并行执行。
  \item 批量ACK测试：模拟一批成功消息，验证ACK请求按stream聚合，并按照\texttt{ack\_batch\_size}批量确认，避免逐条\texttt{XACK}。
  \item PEL恢复测试：模拟待处理消息列表中存在旧消息，验证reclaim worker可以独立执行，并且不会阻塞新消息主消费循环。
  \item 分块预留测试：构造大群接收者列表，验证update id按block连续预留，本地再分配给接收者，从而减少远程MongoDB写入次数。
  \item outbox聚合测试：验证chunk状态变化后，queued、completed、failed和job id等聚合字段能够增量更新，并可通过reconcile路径修复。
\end{enumerate}

对应的工程验证命令为：

\begin{lstlisting}[language=bash,caption={Go投递服务测试命令}]
cd telegram-go-delivery-consumer
go test ./...
staticcheck ./...
\end{lstlisting}

性能门禁中还提供本地fixture，用于输出\texttt{keyed\_dispatch\_batch\_ack}的请求数、batch size、worker lane数量、p50、p95、p99和batch ack count。该fixture不依赖外部Redis或MongoDB，适合在CI中作为回归检查；真实基础设施压测则通过可选infra gate连接Redis、MongoDB和WebSocket环境后执行。

\section{C++图计算内核测试}

C++侧测试重点是验证图快照构建、查询正确性和高出度场景下的性能稳定性。测试设计如下：

\begin{enumerate}
  \item 快照构建测试：构造包含多种边类型和权重的图数据，验证字符串id可以正确intern为整数编号，CSR offset和邻居数组边界正确。
  \item 直接邻居测试：查询某个用户的一阶邻居，验证返回结果、排序和limit限制符合预期。
  \item overlap共同关系测试：构造两个高出度用户，验证共同邻居结果正确，同时检查streaming top-K不会全量物化无关候选。
  \item ranked high-degree测试：对高出度节点进行ranked top-K查询，验证得分排序、时间衰减和预算控制生效。
  \item 多跳遍历测试：构造多层关系图，验证遍历不会超过预算，并能返回更高价值的候选路径。
  \item 快照发布测试：验证新快照发布后查询读取新版本，刷新过程中旧快照仍可服务读请求。
\end{enumerate}

对应的工程验证命令为：

\begin{lstlisting}[language=bash,caption={C++图计算服务测试和基准命令}]
cmake -S telegram-cpp-graph-service \
  -B /tmp/telegram-cpp-graph-build \
  -DCMAKE_BUILD_TYPE=RelWithDebInfo
cmake --build /tmp/telegram-cpp-graph-build -j4 \
  --target graph-store-tests graph-store-bench
ctest --test-dir /tmp/telegram-cpp-graph-build \
  --output-on-failure
cd /tmp/telegram-cpp-graph-build
./graph-store-bench --json
\end{lstlisting}

C++基准输出采用JSON格式，主要包含快照发布、直接高出度查询、ranked top-K、多跳预算遍历、桥接遍历和overlap查询等测试项。报告字段包括吞吐量、p50、p95、p99、scanned、visited、candidates、budget exhausted和memory estimate bytes。这些字段可以帮助判断图服务是否因为快照过大、高出度节点或overlap查询而出现尾延迟风险。

\section{统一性能与发布验证}

除单服务测试外，项目还提供统一性能门禁脚本。该脚本汇总Go投递服务、Rust网关与推荐服务、C++图计算服务的本地fixture和基准结果，统一输出\texttt{summary.json}和\texttt{report.md}。报告字段包括request count、batch size、queue depth、p50、p95、p99、timeouts、fallback、budget exhausted、cache hit和cache miss。对于课程报告而言，该门禁体现了分布式系统中“代码正确性”和“性能回归可观察性”并重的思想。

部署后验证则以服务健康检查为主。VPS上各服务通过Docker Compose运行，关键健康检查包括：

\begin{lstlisting}[language=bash,caption={生产服务健康检查命令}]
curl https://api.xuziqi.tech/health
ssh root@159.203.142.13 '
for port in 5000 4000 4100 4200 4300; do
  curl --silent --show-error --max-time 10 \
    "http://127.0.0.1:${port}/health"
done'
\end{lstlisting}

其中\texttt{5000}对应业务后端，\texttt{4000}对应网关，\texttt{4100}对应Go投递服务，\texttt{4200}对应推荐服务，\texttt{4300}对应C++图计算内核。通过这些端口可以确认多语言服务在同一部署环境下是否均已启动。

\section{测试结论}

根据Apifox联调界面、Supabase数据库结构、推荐算法流程检查、Go投递链路测试和C++图计算基准，可以认为本项目具备课程设计要求的主要测试证据：接口路径明确、请求参数可解析、云端数据库真实存在、推荐算法流程完整、核心数据表能够支撑用户关系和内容行为分析，异步投递链路和图查询内核也具备可验证的正确性与性能观测能力。

同时，测试中发现Supabase公开模式下多张表尚未启用RLS，其中\texttt{users}表还包含\texttt{password}字段。该问题不影响课程报告的结构撰写，但属于真实上线前必须处理的安全风险。建议在正式部署前启用RLS，并将密码字段改为不可逆哈希存储，禁止前端直接访问敏感表。
